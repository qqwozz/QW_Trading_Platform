# QW_Trading_Platform

# Технические требования

## Общая архитектура

Система должна быть построена по принципам микросервисной архитектуры.

Каждый сервис отвечает только за свою предметную область и может развертываться независимо.

Основные сервисы:

* User Service
* Account Service
* Order Service
* Matching Engine
* Market Data Service
* Analytics Service
* API Gateway

Все сервисы должны взаимодействовать через строго определенные API-контракты.

---

# Требования к производительности

## Обработка заявок

Система должна обеспечивать:

* прием не менее 1000 заявок в секунду;
* обработку заявок без потери данных;
* корректное исполнение при высокой нагрузке.

---

## Время отклика

Операция размещения заявки:

не более 100 мс.

Получение рыночных данных:

не более 50 мс.

Передача обновлений по WebSocket:

не более 100 мс с момента возникновения события.

---

## Масштабируемость

Система должна поддерживать горизонтальное масштабирование сервисов без изменения бизнес-логики.

При увеличении нагрузки должно быть возможно увеличение количества экземпляров сервисов.

---

# Требования к надежности

## Отказоустойчивость

Перезапуск отдельного сервиса не должен приводить к остановке всей системы.

Система должна корректно восстанавливать работу после сбоя.

---

## Сохранность данных

Все критически важные данные должны храниться в постоянном хранилище.

Критическими считаются:

* пользователи;
* счета;
* ордера;
* сделки;
* позиции.

---

## Логирование

Все операции должны журналироваться.

Логируются:

* создание ордера;
* отмена ордера;
* исполнение сделки;
* изменение баланса;
* ошибки системы.

Каждая запись должна содержать:

* время события;
* идентификатор пользователя;
* идентификатор операции;
* уровень важности.

---

# Требования к безопасности

## Аутентификация

Доступ к системе возможен только для зарегистрированных пользователей.

Поддерживается:

* регистрация;
* вход в систему;
* обновление токенов доступа.

---

## Авторизация

Пользователь может получать доступ только к собственным данным.

Запрещается:

* просмотр чужих ордеров;
* просмотр чужих позиций;
* изменение чужих данных.

---

## Защита API

Все запросы должны выполняться через HTTPS.

Поддерживаются:

* JWT Access Token;
* Refresh Token.

---

## Ограничение запросов

Для публичных API должна быть реализована защита от чрезмерного количества запросов.

---

# Требования к данным

## База данных

Система должна использовать реляционную СУБД.

Требования:

* транзакционность;
* поддержка индексов;
* поддержка репликации;
* поддержка резервного копирования.

---

## Исторические данные

История операций не должна удаляться.

Все сделки должны храниться бессрочно.

---

## Аудит

Для каждой сделки должна быть возможность восстановить:

* покупателя;
* продавца;
* цену исполнения;
* объем;
* время исполнения.

---

# Требования к API

## REST API

Для клиентских операций используются REST API.

Поддерживаются операции:

* регистрация;
* авторизация;
* получение портфеля;
* размещение ордера;
* отмена ордера;
* получение истории.

---

## Документация

Все API должны быть документированы.

Документация должна генерироваться автоматически.

---

# Требования к передаче рыночных данных

## WebSocket

Рыночные данные должны передаваться через постоянное соединение.

Поддерживаются подписки на:

* сделки;
* изменения книги заявок;
* изменения цены;
* статистику инструмента.

---

## Реальное время

При изменении состояния рынка данные должны автоматически доставляться подписанным клиентам.

---

# Требования к аналитике

Система должна рассчитывать:

* прибыль и убыток;
* доходность;
* коэффициент Шарпа;
* максимальную просадку;
* процент прибыльных сделок.

Расчеты должны выполняться независимо от торгового ядра.

---

# Требования к контейнеризации

Каждый сервис должен поставляться как отдельный контейнер.

Контейнер должен содержать:

* приложение;
* необходимые зависимости;
* конфигурацию запуска.

---

# Требования к оркестрации

Система должна поддерживать развертывание в Kubernetes.

Для каждого сервиса должны быть определены:

* Deployment;
* Service;
* ConfigMap;
* Secret;
* Health Checks.

---

# Требования к мониторингу

Система должна предоставлять метрики:

* количество ордеров;
* количество сделок;
* время обработки запросов;
* количество ошибок;
* использование ресурсов.

---

# Требования к тестированию

## Unit-тесты

Покрытие бизнес-логики не менее 80%.

---

## Интеграционные тесты

Должны проверяться:

* размещение ордеров;
* исполнение ордеров;
* обновление балансов;
* генерация сделок.

---

## Нагрузочные тесты

Система должна проходить сценарий:

* 10 000 пользователей;
* 50 000 ордеров;
* непрерывная торговля в течение 30 минут.

Фиксируются:

* средняя задержка;
* P95;
* P99;
* количество ошибок.

---

# Требования к наблюдаемости

Система должна поддерживать:

* централизованные логи;
* метрики;
* трассировку запросов между сервисами.

Для любого пользовательского действия должен быть доступен полный путь прохождения запроса через систему.


# Архитектурные решения и компромиссы

## Общие принципы

При проектировании системы основными целями являлись:

* минимизация задержек при обработке ордеров;
* обеспечение консистентности торговых данных;
* горизонтальная масштабируемость;
* независимое развитие компонентов системы;
* отказоустойчивость.

Для достижения этих целей были приняты следующие архитектурные решения.

---

# Выделение Matching Engine в отдельный сервис

## Решение

Торговое ядро выделено в отдельный сервис и не объединяется с API или бизнес-сервисами.

---

## Причины

Торговый движок является наиболее критичным компонентом системы.

Он отвечает за:

* обработку заявок;
* ведение книги ордеров;
* поиск встречных заявок;
* исполнение сделок.

Любая дополнительная нагрузка может увеличивать задержки исполнения.

Изоляция позволяет:

* независимо масштабировать остальные сервисы;
* исключить влияние пользовательских запросов;
* контролировать производительность торгового ядра.

---

## Компромисс

Увеличивается сложность инфраструктуры.

Появляются:

* межсервисные вызовы;
* дополнительная сеть;
* необходимость синхронизации данных.

---

# Хранение книги заявок в памяти

## Решение

Книга заявок хранится в оперативной памяти Matching Engine.

---

## Причины

Основная задача торгового движка — минимальная задержка исполнения.

Работа с базой данных при каждом изменении книги заявок приводит к значительным задержкам.

Хранение в памяти обеспечивает:

* быстрый поиск встречных заявок;
* минимальное время вставки;
* минимальное время удаления.

---

## Компромисс

При аварийном завершении процесса состояние книги заявок может быть потеряно.

Для минимизации риска все события дополнительно записываются в журнал событий.

После перезапуска книга может быть восстановлена путем воспроизведения событий.

---

# Использование событийной архитектуры

## Решение

После каждого значимого действия публикуется событие.

Примеры:

* OrderCreated
* OrderCancelled
* TradeExecuted
* BalanceChanged

---

## Причины

Сервисы не должны зависеть друг от друга напрямую.

Событийная модель позволяет:

* уменьшить связанность;
* упростить масштабирование;
* добавлять новые сервисы без изменения существующих.

---

## Компромисс

Появляется eventual consistency.

Некоторые сервисы могут получать изменения с небольшой задержкой.

---

# Разделение торгового контура и аналитики

## Решение

Аналитические расчеты вынесены в отдельный сервис.

---

## Причины

Расчет аналитических показателей может требовать значительных ресурсов.

Например:

* расчет Sharpe Ratio;
* расчет просадок;
* построение статистики.

Эти операции не должны влиять на скорость исполнения ордеров.

---

## Компромисс

Возникает необходимость синхронизации аналитических данных.

---

# Использование WebSocket для рыночных данных

## Решение

Передача рыночных данных осуществляется через постоянное соединение.

---

## Причины

Рынок постоянно изменяется.

Периодический опрос сервера создает:

* лишнюю нагрузку;
* дополнительные задержки.

WebSocket позволяет доставлять обновления сразу после возникновения события.

---

## Компромисс

Усложняется управление подключениями клиентов.

Необходимо контролировать:

* количество соединений;
* переподключения;
* потерю соединения.

---

# Использование PostgreSQL как основной базы данных

## Решение

Исторические данные сохраняются в PostgreSQL.

---

## Причины

Требуются:

* транзакционность;
* консистентность;
* надежное хранение истории.

PostgreSQL обеспечивает ACID-гарантии и подходит для хранения финансовых данных.

---

## Компромисс

Запись большого объема данных может становиться узким местом.

Для масштабирования в будущем возможно использование реплик чтения и партиционирования.

---

# Использование горизонтального масштабирования

## Решение

Все сервисы, кроме торгового ядра, проектируются с возможностью горизонтального масштабирования.

---

## Причины

Количество пользователей может расти независимо от количества торговых инструментов.

Горизонтальное масштабирование позволяет увеличивать пропускную способность без изменения бизнес-логики.

---

## Компромисс

Появляются задачи:

* балансировки нагрузки;
* управления состоянием;
* синхронизации данных между экземплярами.

---

# Использование API Gateway

## Решение

Внешние клиенты работают только через единую точку входа.

---

## Причины

Это позволяет централизовать:

* авторизацию;
* маршрутизацию;
* аудит;
* ограничение запросов.

---

## Компромисс

API Gateway становится критически важным компонентом.

Для высокой доступности требуется запуск нескольких экземпляров.

---

# Выбор модели консистентности

## Решение

Для торгового ядра используется строгая консистентность.

Для аналитики и отображения данных допускается eventual consistency.

---

## Причины

Исполнение ордеров требует абсолютной точности.

Аналитические данные могут обновляться с небольшой задержкой без влияния на корректность торговли.

---

## Компромисс

Пользователь может временно видеть устаревшие аналитические показатели.

Однако торговые результаты всегда остаются корректными.

---

# Выбор приоритетов системы

При возникновении конфликта требований система отдает приоритет следующим характеристикам:

1. Корректность исполнения сделок.
2. Сохранность данных.
3. Низкая задержка торгового ядра.
4. Масштабируемость.
5. Удобство разработки.

Ни одно решение не должно ухудшать корректность торговых операций ради производительности или удобства реализации.
